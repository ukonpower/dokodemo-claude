# Research Phase - コードベース調査

タスクに関連するコードベースを**徹底的に**調査し、`.workflow-tools/research.md` に調査結果を永続化する。
この調査結果は後続の計画フェーズの基盤となる。

## ワークフロー

### Step 1: タスクの確認

AskUserQuestion を使って以下を確認する:

- **何を実現したいか**: 具体的なタスク・課題の内容
- **スコープ**: 影響範囲の想定（わかる範囲で）

すでにユーザーが明確にタスクを説明している場合はスキップしてよい。

### Step 1.5: 既存の research.md の処理

`.workflow-tools/research.md` が既に存在する場合:

- **デフォルト（新規作成の指示）**: 既存の research.md を削除してから新たに作成する
- **「更新して」「追記して」等の更新指示があった場合**: 削除せず、既存の内容を読み込んで更新・追記する

ユーザーのプロンプトに「更新」「追記」「反映」「修正」などの表現が含まれる場合は更新モードとして扱う。

**⚠️ 更新モードでの必須手順**: 更新・追記する場合は、**必ず最初に Read ツールで既存の research.md の全内容を読み込んでから** 編集を行うこと。読み込まずに Edit や Write で更新しようとすると失敗する。

### Step 2: 徹底的な調査

以下を**深く、詳細に**調査する。「表面的な理解」ではなく「実装レベルの理解」を目指す:

1. **関連ファイル・シンボルの特定**
   - タスクに直接関係するファイル、クラス、関数を網羅的に探す
   - Grep, Glob, Task(Explore) を活用して漏れなく調査

2. **依存関係の把握**
   - 変更対象が依存している、または依存されているコードを調査
   - インポート関係、呼び出し元/呼び出し先を追跡

3. **既存パターンの理解**
   - コードベースで使われている規約・パターンを把握
   - 命名規則、ディレクトリ構造、エラーハンドリングの方法等

4. **制約・注意点の洗い出し**
   - 変更が壊す可能性のある周辺システム
   - テストカバレッジの状態
   - パフォーマンスやセキュリティへの影響
   - **注意**: 後方互換性の維持は目的としない。根本から変更する前提で、影響範囲を把握することに集中する。

### Step 3: research.md に出力

`.workflow-tools/` ディレクトリが存在しない場合は作成する。
調査結果を `.workflow-tools/research.md` に書き出す。以下のフォーマットを使用:

```markdown
# Research: {タスク名}

## タスク概要
{何を実現したいか}

## 関連ファイル・シンボル
| ファイル | 主要シンボル | 役割 |
|---------|------------|------|
| path/to/file.ts | ClassName, functionName | 説明 |

## 依存関係
- `FileA` → `FileB`: {関係の説明}

## 既存パターン
- {パターン名}: {説明}

## 制約・注意点
- {制約1}
- {制約2}

## 参考になる既存実装
- {既存の類似実装へのポインタ}
```

**⚠️ コード記述のルール**: research.md にソースコードを記載する場合は、**重要な部分のみを抜粋**して記述すること。ファイル全体やクラス全体をコピーしない。理解に必要な型定義、関数シグネチャ、重要なロジック部分だけを簡潔に示す。ファイルが肥大化するのを防ぐため。

### Step 4: 報告と次ステップ案内

調査結果のサマリーをユーザーに報告し、次のステップを案内する:

> 調査が完了しました。`.workflow-tools/research.md` に結果を保存しています。
> 次は「計画を作って」と指示すると、この調査結果を基に実装計画を作成します。

## 重要なルール

- **md ファイルを Edit や Write で変更・作成する前に、必ず Read ツールで現在の内容を読み込むこと。** Write ツールは既存ファイルを事前に Read していないとエラーになる。新規作成・更新を問わず、対象ファイルが存在する可能性がある場合は必ず先に Read する。
- **このフェーズではコードの変更を絶対に行わない。** ファイルの編集・作成・削除は一切禁止。調査と理解のみに集中する。
- 「deeply」「in great detail」の精神で、表面的な調査で終わらせない。
- 周辺システムへの影響を見逃すことが最大の失敗モード。依存関係の調査を怠らない。
- research.md への書き出しのみが唯一許可されるファイル操作である。
