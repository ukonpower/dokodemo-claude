# Implement Phase - 計画の実行

`.workflow-tools/plan.md` に記載された実装計画を実行する。

## ワークフロー

### Step 1: 調査結果と計画の読み込み

1. `.workflow-tools/research.md` が存在する場合は読み込む（調査フェーズの成果物）
2. `.workflow-tools/plan.md` を読み込む。存在しない場合:

> `.workflow-tools/plan.md` が見つかりません。
> 先に「計画を作って」と指示して実装計画を作成してください。

research.md の内容は実装時の背景知識・設計判断の根拠として活用する。

### Step 2: 実装の実行

計画のステップを順番に実行する:

1. 各ステップの実装を行う
2. ステップ完了時に plan.md のチェックリストを更新（`- [ ]` → `- [x]`）
3. 問題が発生した場合はユーザーに報告し判断を仰ぐ

**実装中のルール:**
- **React コードを実装する場合は、`/coding-react` スキルを呼び出してルールを適用する。** コンポーネント、フック、useEffect の実装は coding-react のベストプラクティスに準拠すること。
- 計画に従って実装する。計画にない変更は行わない。
- 型チェックやリントがある場合は適宜実行して品質を維持する。
- 不要なコメントは追加しない。コードは自己説明的に書く。
- 厳密な型指定を維持する（`any` 等の緩い型は避ける）。
- **シンプルで簡潔な実装にする。** 後方互換性のためのシム、非推奨ラッパー、旧コードの温存は行わない。根本から変更し、不要になったコードは削除する。

### Step 3: 品質確認

実装完了後、以下を確認:

- 型チェック通過（TypeScript の場合 `tsc --noEmit` 等）
- リント通過（ESLint, Biome 等、プロジェクトのツールに従う）
- 既存テストが通る（テストがある場合）
- 計画のチェックリストがすべて `[x]` になっている

### Step 4: 完了報告

> 実装が完了しました。
>
> **完了したステップ:**
> {チェックリストのサマリー}
>
> **品質確認:**
> - 型チェック: {結果}
> - リント: {結果}
> - テスト: {結果}
>
> `.workflow-tools/plan.md` に進捗を反映しています。

## 実装中のフィードバック対応

ユーザーからの実装中の指示は短く具体的であることが多い:

- **単文での補正** → 即座に対応して実装を継続
- **スクリーンショット** → UI の確認・修正に反映
- **既存コードへのポインタ** → 参考にして実装を修正

これらの軽微な修正は plan.md の更新なしで直接対応してよい。

## 重要なルール

- **md ファイルを Edit や Write で変更・作成する前に、必ず Read ツールで現在の内容を読み込むこと。** Write ツールは既存ファイルを事前に Read していないとエラーになる。新規作成・更新を問わず、対象ファイルが存在する可能性がある場合は必ず先に Read する。
- plan.md が存在しない状態で実装を開始しない。
- 計画にない大きな変更が必要になった場合は、実装を中断してユーザーに相談する。
- すべてのステップが完了するまで自律的に作業を継続する。途中で不要な確認を挟まない。
