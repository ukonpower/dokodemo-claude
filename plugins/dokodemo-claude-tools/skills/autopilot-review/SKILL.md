# autopilot-review

autopilot（無人開発ループ）の実施後に、ログと成果物を分析してふりかえりを行うスキル。「何タスク進んだか」ではなく「実物がどこまで良くなったか」を軸に評価し、autopilot.md の改訂まで行う。

## Workflow

### Step 1: 記録の収集

以下を読み、ループの実態を時系列で再構成する:

- `git log --format='%h %ad %s' --date=format:'%m-%d %H:%M'` — サイクルの時刻分布・ペース・タスクの種別比率（feat/fix/chore）
- `docs/journal.md`・`docs/journal-archive.md` — 各サイクルの実際の作業と詰まった点
- `docs/tasks.md`・`docs/tasks-archive.md` — BLOCKED の履歴・タスク分割の頻発箇所・受け入れ条件の書かれ方
- `docs/insights.md` — ループ自身が自覚していた問題
- `docs/decisions.md` — ゴール設定の判断と、人間の介入があったか
- `docs/autopilot/shots/` — スクリーンショットの git 履歴（見た目の変化を時系列で確認）

### Step 2: 成果物の実地確認

ログを読むだけで済ませない。実際に起動し、autopilot.md に記載のターゲット環境で**初見の利用者として**垂直スライスを通しで触る。「操作できるか」「初見で次の操作が分かるか」「到達ラインまでの距離」を自分の目で確認する。ここでの体感とログ上の進捗の乖離が、ふりかえりの最重要データになる。

### Step 3: 分析

以下の観点でログと実地確認を突き合わせる。各観点は過去の実ループで実際に起きた失敗パターン:

1. **完了と実物の乖離**: DONE の件数に対して、実物はどこまで動くか。「ユニットテストのみで完了扱いになったタスク」「配線されないまま積まれたコード」がないか
2. **致命的バグの生存期間**: 実物を壊すバグについて「混入したサイクル→発見されたサイクル」を特定する。長期生存していたなら、どのゲートが素通りさせたか
3. **検証の形骸化**: 「動作確認」タスクが実際には何を確認したか。リアルタイム検証の失敗を状態注入等で代替した場合、代替によって検証されなくなったもの（操作感・タイミング）は何か
4. **垂直スライス vs 横展開の時間配分**: コミット時刻から各マイルストーンの所要を出し、実物が壊れている期間に幅の拡張へ時間が使われていなかったか
5. **設計目標と検証環境のズレ**: ゴール文書の設計目標（例: モバイルファースト）どおりの環境で検証されていたか
6. **サイクルの浪費**: 同じ失敗の繰り返し・BLOCKED の長期滞留・過剰なタスク分割・メンテナンスの空回りがないか
7. **手順書の遵守と例外**: autopilot.md の規定が守られたか。守られなかった箇所は「モデルの怠慢」ではなく「手順書の設計不備」として扱う

### Step 4: レポートと改訂

1. **ふりかえりレポート**を提示する: 結論（実物の到達度）→ 事実（ログからの根拠、サイクル番号・時刻付き）→ 根本原因 → 改善案、の順。良かった点（機能した仕組み）も明記し、次回も維持する
2. **autopilot.md の改訂案**を具体的な diff の形で提示し、ユーザー確認の上で適用する
3. 改善が手順書で解決できない種類のもの（テスタビリティフック不足・スモークの検査項目不足）なら、tasks.md へ最優先タスクとして起票する
4. 教訓が汎用的なら、dokodemo-claude リポジトリ同梱のテンプレート（`plugins/dokodemo-claude-tools/skills/autopilot-setup/references/autopilot-template.md`）への還元も提案する（編集時は `plugins/.claude-plugin/marketplace.json` と `plugins/dokodemo-claude-tools/.claude-plugin/plugin.json` のバージョンも揃えてバンプする）

### Step 5: 記録

- ふりかえりの結論と「次回どうすべきか」を auto-memory に保存する（1ファイル1トピック、既存メモリがあれば更新）
- 次のループを続行する場合は、レポートで挙げた最優先の修正が tasks.md の先頭に積まれていることを確認してから終える

## 注意事項

- 評価軸は常に「実物を触った体験」であり、タスク消化数・コミット数・テスト件数を成果として扱わない
- ループ中のモデルの個々の作業品質（真因特定・記録の丁寧さ等）と、仕組みの設計品質（ゲート・DoD・順序）を分けて評価する。前者が良くても後者が悪ければ結果は出ない
