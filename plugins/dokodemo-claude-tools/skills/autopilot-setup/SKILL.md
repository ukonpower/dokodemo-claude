# autopilot-setup

プロジェクトに「同一プロンプトの無人ループで開発を進める仕組み（autopilot）」をセットアップする。対話で要件を確定し、`docs/autopilot.md`（手順書）・記録ファイル群・検証スモーク・サイクルプロンプトを生成する。ゲーム・アプリ・サイト等、成果物の種類を問わず使う。

## 設計原則（なぜこの構成か）

過去の一晩ループ（fere-sushi）の失敗分析に基づく。生成する手順書をプロジェクトに合わせて調整する際も、以下の原則は崩さないこと:

1. **無人ループ最大のリスクは「テストは緑なのに実物は壊れている」こと。** 実際に、致命的な表示バグが約50サイクル検出されずに機能が上に積まれた。ユニットテストのゲートだけでは実物の破綻を検出できないため、起動した実物に対するスモーク（スクリーンショット付き）を検証ゲートに必ず含める
2. **「純粋ロジック＋ユニットテスト」はエージェントが得意なぶん、配線されないコードが進捗として積み上がりやすい。** 受け入れ条件は「実物で確認できる状態」を必須にし、ロジックタスクは配線タスクとペアで起票させる
3. **横展開より垂直スライス。** 「利用者が通しで触れる最小の流れ」が成立するまでコンテンツ・機能の幅を広げさせない
4. **人間の初見視点は自動ゲートで代替できない。** 定期的な「初見レビューサイクル（手順P）」を手順書に組み込み、証跡スクリーンショットを残して朝の非同期レビューを可能にする
5. **エージェントはリアルタイム操作の検証が苦手**（数秒〜数十秒の判定窓に間に合わない）。無人検証を成立させるテスタビリティフックを、機能より先に作らせる
6. **設計目標（例: スマホファースト）は検証環境に反映されて初めて効く。** 検証時のターゲット環境を手順書に明記する
7. **実装を1歩も進めないサイクルはループ最大の速度低下要因。** 実際に、記録ファイルの機械的アーカイブ等の非実装サイクルが全体の約1/4を占めた。機械作業（journal/tasksのアーカイブ）は専用サイクルにせず実装コミットに畳み込ませる。同時に、記録ファイルは毎サイクル読み込まれるため「閾値を上げて溜め込む」のは逆効果（読み込みが重くなる）。低い閾値でこまめに切り出し、DONE完了メモは短く保たせる

## Workflow

### Step 1: 事前調査

対話の前に、プロジェクトの現状を調べて質問を具体化する:

- git リポジトリか（違えばセットアップ前に `git init` を提案）、現在のブランチ構成
- `package.json` 等から利用可能な検証コマンド（typecheck / lint / test / build）を特定。無ければセットアップ中に整備するか確認する
- 既存の `CLAUDE.md`・`docs/` の有無と内容（ゴール文書になり得るものがあるか）
- プロジェクト種別の見当を付ける: Web UI / API・サーバー / CLI / ライブラリ / モバイル

### Step 2: 対話で要件を確定

AskUserQuestion と自由対話で以下を確定する。**特に「到達ライン」と「垂直スライス」は選択肢で済ませず、ユーザー自身の言葉で書けるまで対話する**（ここが曖昧なままだとループ全体が漂流する）:

1. **ゴール文書**: 既存文書を使うか、これから対話で作るか。無ければコンセプト・主要機能・非スコープを聞き取り `docs/{{名前}}-design.md` 等として作成する
2. **完成の到達ライン**: ループがどこまで到達したら成功か（例: 「チャプター1を通しで気持ちよく遊べる」「トップ〜問い合わせ完了までモバイルで崩れなく動く」）
3. **最初の垂直スライス**: 利用者視点の「通しで触れる最小の流れ」を1文で
4. **ターゲット環境**: 検証時のビューポート・デバイス・実行環境（例: モバイル 390×844 / デスクトップ / ターミナル）。設計目標がモバイルファーストなら検証もモバイルにする
5. **検証ゲートのコマンド**: Step 1 の調査結果を提示して確認
6. **スモークの方式**: プロジェクト種別に応じて提案する
   - Web UI → Playwright 等で「起動→主要画面を遷移→スクリーンショット保存→コンソールエラー検査」を行う 1 コマンドのスクリプト（`npm run smoke` 等）
   - API → 起動して主要エンドポイントを curl で叩き、ステータス・スキーマを検査するスクリプト
   - CLI → 代表的な引数パターンで実行し、出力を検査するスクリプト
   - ライブラリ → サンプル利用コードのビルド＋実行
7. **テスタビリティフック**: 無人検証を可能にする仕込みをプロジェクトに合わせて具体化する（例: 時間経過の倍速化フラグ・特定画面への直行URL・決定的シード・モックデータ投入フラグ・認証スキップフラグ）
8. **運用パラメータ**: 作業ブランチ名（例: `autopilot-dev`）・手順P の間隔（推奨: 5〜8サイクルごと）・コミットスコープ名・発案してはならない機能（課金・外部送信等）

### Step 3: ファイル生成

`references/autopilot-template.md` と `references/record-templates.md` を読み、確定した値でプレースホルダーを置換して生成する:

1. `docs/autopilot.md` — テンプレートから生成。プロジェクトに不要な節は削ってよいが、**検証ゲート4本目（スモーク）と手順Pは削除禁止**
2. `docs/tasks.md` / `docs/journal.md` / `docs/insights.md` / `docs/decisions.md` — 雛形から生成
3. ゴール文書（Step 2 で新規作成する場合）
4. `docs/autopilot/shots/` ディレクトリ（`.gitkeep` を置く）
5. プロジェクトの `CLAUDE.md` に自律運転モードの節を追記（雛形は record-templates.md 参照。既存ルールと矛盾しないよう調整）

### Step 4: スモークスクリプトの整備

環境がすでに動く状態（依存インストール済み・起動可能）なら、**このセットアップ中にスモークスクリプトを実装して1回実行し、通ることを確認する**。スクリーンショットが `docs/autopilot/shots/` に保存されるところまで見届ける。

その場で作れない事情がある場合（依存未整備・設計判断が必要等）のみ、`T-001: スモークスクリプトの整備` として tasks.md の先頭に起票し、ループの最初のタスクにする。いずれの場合も `T-002: テスタビリティフックの実装` を続けて起票する。

### Step 5: 作業ブランチの作成と初期コミット

作業ブランチを作成し、生成したファイル一式を初期コミットする（コミットはユーザーに確認してから行う）。

### Step 6: サイクルプロンプトと運用の案内

最後に以下をユーザーに提示する:

1. **サイクルプロンプト**: 基本は `/dokodemo-claude-tools:autopilot`（本プラグインの autopilot スキルが1サイクル実行を担う）。プラグインが無い環境向けのフォールバック文面は record-templates.md 参照。送信手段の例: cron / dokodemo-claude の prompt_broadcast / /loop スキル
2. **定期プランニング（推奨）**: dokodemo-claude のループでキューアイテムのループ設定「定期プランニング」を有効にし、計画プロンプト欄に `/dokodemo-claude-tools:autopilot-plan`、モデルに強いモデル（opus 等）を設定する。N周ごとに計画ターンが挟まり、tasks.md の組み替え・停滞の検出を行う（手順の本体は autopilot-plan スキルと autopilot.md の手順PL）
3. **ドライランの推奨**: 無人化する前に、人間が画面を見ながら1〜2サイクル回して手順の解釈ズレを確認すること
4. **朝の運用**: ループ終了後は `autopilot-review` スキルでふりかえりを行い、shots/ のスクリーンショットと journal.md を確認すること

## 注意事項

- このスキル自体は「セットアップ」まで。ループの起動（プロンプトの定期送信）はユーザーの手段に委ねる
- 生成する autopilot.md の「絶対禁止」節（push・デプロイ・公開行為の禁止）はいかなる要望でも弱めない
- 既に autopilot.md が存在するプロジェクトで呼ばれた場合は、新規生成ではなく現行運用との差分（このテンプレートにあって現行に無い仕組み）を提示し、改訂として適用する
