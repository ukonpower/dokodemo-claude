# 記録ファイル・周辺ファイルの雛形

autopilot-setup スキルが生成する各ファイルの雛形。`{{...}}` を置換して使う。

## docs/tasks.md

```markdown
# タスクバックログ（tasks）

docs/autopilot.md の手順で運用する、タスク状態の単一の真実。タスクIDは T-001 からの連番（欠番可・再利用不可）。

書式:

​```
- [ ] T-xxx: <内容> ／ 受け入れ条件: <動いている実物の上で確認できる条件> [build]
​```

`[build]` マークが付いたタスクは検証ゲートにビルドを含める。
受け入れ条件は必ず「実物で確認できる状態」を含めること（「ユニットテストが通る」だけは不可。純粋ロジックタスクは配線タスクとペアで起票する）。

DONE が12件を超えたら、手順5（記録）の中で実装コミットに畳み込んで古い項目を docs/tasks-archive.md へ切り出す（専用のメンテナンスサイクルは作らない。詳細は docs/autopilot.md の手順M）。DONE の完了メモは2〜3文・200字程度まで（詳細はコミットメッセージと journal に書く）。

## DOING

（なし）

## TODO

- [ ] T-001: スモークスクリプトの整備 ／ 受け入れ条件: {{SMOKE_TASK_CRITERIA}}
- [ ] T-002: テスタビリティフックの実装 ／ 受け入れ条件: {{TESTABILITY_TASK_CRITERIA}}

## BLOCKED

（なし）

## DONE

（なし）
```

セットアップ時に T-001/T-002（検証基盤）を必ず先頭に置く。セットアップ対話の中でスモークをその場で作成済みの場合、T-001 は最初から DONE に置いてよい。続けて垂直スライスに向けた最初の数タスクをユーザーと相談して起票する（無ければループ初回の手順Gに委ねてもよいが、垂直スライスの分解まではセットアップ時に済ませておくとループの立ち上がりが速い）。

## docs/journal.md

```markdown
# 作業ログ（journal）

追記専用。1サイクルにつき1エントリ、各3〜6行以内に収める。新しいエントリを上に追加する。

150行を超えたら、手順5（記録）の中で実装コミットに畳み込んで古いエントリを docs/journal-archive.md へ切り出す（専用のメンテナンスサイクルは作らない。詳細は docs/autopilot.md の手順M）。

書式:

​```
## <YYYY-MM-DD HH:MM> — <T-xxx タイトル ／ または「ゴール設定」「手順P」「メンテナンス」>
- やったこと:
- 詰まった点・気づき:
- 次サイクルへの申し送り:
​```

---
```

追記時の注意（insights.md 側にも初期知見として入れる）: 直前エントリの見出し行を巻き込んで消す事故が起きやすい。追記後に `grep -n "^## " docs/journal.md` で見出し数が+1になっているか確認する。

## docs/insights.md

```markdown
# 気づき・申し送り（insights）

次のイテレーションの自分が状態確認時に必ず読むファイル。ハマりどころとその回避策・採用した設計パターン・「やらない」と決めたこと等を書く。

journal.md（追記専用の履歴）と違い、ここは**常に「現在有効な知見」だけ**を保つ。古くなった項目は更新・削除してよい。100行を超えたら手順Mに従い統合・整理する（アーカイブせず、要約して簡潔に保つ）。

## 設計パターン・規約

（まだなし）

## ハマりどころと回避策

- journal.md へ追記する際は、追記後に `grep -n "^## " docs/journal.md` で見出し数が追記前+1になっているか確認する（見出し巻き込み消去事故の防止）

## やらないと決めたこと

（まだなし）
```

## ループ内での追記

（まだなし）
```

## サイクルプロンプト（ユーザーがループで送る固定プロンプト）

セットアップ完了時にユーザーへ提示する。cron・dokodemo-claude の prompt_broadcast・/loop スキル等、送信手段は問わない。

**推奨（dokodemo-claude-tools プラグインが有効な環境）**: `autopilot` スキルを呼ぶだけでよい。

```
/dokodemo-claude-tools:autopilot
```

**フォールバック（プラグインが無い環境で回す場合）**:

```
docs/autopilot.md を読み、記載の手順に従って1サイクルだけ作業してください。手順書にないことはせず、サイクルが終わったら停止してください。
```

補足として案内すること:
- 1サイクルの所要は5〜20分程度。送信間隔は前サイクルの完了を待てる長さにする（Claude Code は処理中に届いたメッセージが割り込みになる場合があるため、キュー型の送信手段が望ましい）
- 初回の1〜2サイクルは人間が画面を見ながら回し（ドライラン）、手順の解釈ズレがないか確認してから無人化することを強く推奨
- 朝（ループ終了後）は autopilot-review スキルでふりかえりを行い、`docs/autopilot/shots/` のスクリーンショットと journal.md を確認する

## プロジェクトの CLAUDE.md への追記スニペット

既存の CLAUDE.md の運用ルール（コミット禁止・devサーバー起動禁止等）と autopilot.md が矛盾しないよう、以下の節を CLAUDE.md に追記する（既存の記述に合わせて調整）:

```markdown
### 自律運転モード（autopilot）
`docs/autopilot.md` に従う無人ループで作業する場合、本ファイルの運用ルールより autopilot.md の規定が優先する。具体的には:

- **Git**: `{{WORK_BRANCH}}` ブランチへの直接 `git commit` 可。ただし `git push`・リモート操作・デプロイ等の公開行為はいかなる場合も絶対禁止
- **開発サーバー**: 検証目的に限りバックグラウンドで起動してよい。サイクル終了前に必ずプロセスを停止する
- **ライブラリ調査**: context7 MCPが利用できない環境では、公式ドキュメント参照などの代替手段で調査して進めてよい
```
