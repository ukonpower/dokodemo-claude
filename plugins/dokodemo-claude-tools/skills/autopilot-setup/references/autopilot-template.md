autopilot-setup スキルがプロジェクトの `docs/autopilot.md` を生成する際の雛形。`{{...}}` プレースホルダーを対話で確定した値に置換し、`<!-- SETUP: ... -->` コメントは指示に従って解決した上で削除すること。プロジェクトに不要な節（例: UIを持たないCLIツールにおけるビューポート指定）は丸ごと削ってよいが、**検証ゲートの4本目（スモーク）と手順P（初見レビュー）は削除禁止**（過去の失敗の再発防止のための核。理由はSKILL.mdの設計原則を参照）。

# 自律運転手順書（autopilot）

このプロジェクトは同一プロンプトの無人ループで開発される。あなた（Claude）はこの手順書に従い、**1回の実行で1サイクルだけ**作業する。進捗を見る人間はいない。すべての状態はリポジトリ内のファイル（docs/tasks.md・docs/journal.md・docs/insights.md・git履歴）に残すこと。

## 絶対禁止（最優先ルール・他のすべてに優先する）

- **公開に類する行為の一切**: `git push`／リモートの追加・変更／`gh` や GitHub API の操作／デプロイ（Vercel・Netlify・Cloudflare 等）／`npm publish`／外部URLを生成する行為／外部サービスへの投稿・送信
- ネットワーク利用は「パッケージマネージャでの依存取得」と「ライブラリドキュメントの参照（context7 等）」のみ許可
- git の破壊的操作（`reset --hard`・force 系・ブランチ削除・履歴改変）
- {{MAIN_BRANCH}} ブランチへのコミット（作業はすべて `{{WORK_BRANCH}}` ブランチ上で行う）
- サーバープロセスを起動したままサイクルを終えること（検証目的での起動は可。ただし必ずバックグラウンドで起動し、サイクル終了前に停止する）
- リポジトリ外のファイル変更（auto-memory への記録は除く）

## このプロジェクトの到達ライン

- **ゴール文書**: {{GOAL_DOC}}（何を作るかの単一の真実。迷ったらこれに立ち返る）
- **完成の到達ライン**: {{DONE_DEFINITION}}
- **最初の垂直スライス**: {{VERTICAL_SLICE}}
  - 垂直スライスとは「利用者が {{TARGET_ENV}} で最初から最後まで通しで触れる最小の一連の流れ」のこと
  - **垂直スライスが手順Pで合格するまで、機能の横展開（コンテンツ追加・画面追加・オプション機能）をゴールに設定してはならない**

## 1サイクルの手順

### 1. 状態確認
- 現在のブランチが `{{WORK_BRANCH}}` であることを確認する（違えば checkout する）
- `git status` を確認する。未コミットの変更が残っていれば前回サイクルの中断とみなし、tasks.md の DOING と突き合わせて復旧から始める
- docs/tasks.md を読む
- docs/insights.md に目を通す（申し送りの反映）
- docs/insights.md が肥大化の基準を超えていれば、実装より先に手順M（insights統合サイクル）を行う。journal・tasks の肥大化チェックはここでは行わない（手順5で記録を追記するときにまとめて確認・対応する）
- journal.md の直近エントリから、前回の手順P実施サイクルを確認する。**{{REVIEW_INTERVAL}}サイクル以上、手順Pを実施していなければ、実装より先に手順P（初見レビューサイクル）を行う**

### 2. 分岐
- **DOING のタスクがある** → その続きを再開する（手順3へ）
- **TODO のタスクがある** → 最上位を1つ選び、DOING に移動してから着手する（手順3へ）
- **TODO が空** → 「ゴール設定サイクル」を行う（手順G）。このサイクルでは実装しない

### 3. 実装
- CLAUDE.md の規約に従う
- 新しいライブラリを導入する場合は context7 等でドキュメントを確認してから実装する
- 1サイクルで完了させるのは選んだ**1タスクのみ**が原則。関係ない改善に手を広げない（気づきは insights.md か新規TODOとして記録するに留める）
- **例外（極小・関連タスクの束ね）**: 同一モジュール内で、かつ新規の設計判断を伴わない些末なタスク（例: 同種のアセット差し替えを数種、ロジックとその配線）は、合計しても1サイクルに収まるなら1サイクルでまとめて完了してよい（検証ゲートは1回で共有する）。束ねる場合は着手前に対象タスクを**すべて DOING へ移す**（中断時の復旧が DOING との突き合わせで行われるため）。判断に迷う・設計を選ぶ余地があるものは束ねず1タスクに留める
- 着手後に「1サイクルで終わらない大きさ」と判明したら、実装を中断してタスクを tasks.md 上で分割し、その旨を記録して終了してよい

### 4. 検証ゲート（すべて緑になるまでタスクを完了扱いにしない）

1. {{GATE_TYPECHECK}}
2. {{GATE_LINT}}
3. {{GATE_TEST}}
4. **スモーク**: {{GATE_SMOKE}}
5. タスクに `[build]` マークがあれば {{GATE_BUILD}} も実行する

1〜3は `{{GATE_TYPECHECK}} && {{GATE_LINT}} && {{GATE_TEST}}` のように連結して1回の実行で済ませてよい。

スモーク（4）は**コード変更を伴う全サイクルで必須**。ユニットテストが緑でも、起動した実物の上で壊れていることは検出できない（見た目の崩れ・配線漏れ・レイアウト破綻はテストを素通りする）。スモークは以下を満たすこと:

- 1コマンドで完走する（対話的操作を要しない）
- {{SMOKE_DETAIL}}
- スクリーンショット等の証跡は `docs/autopilot/shots/` に**固定ファイル名で上書き保存**する（git履歴が見た目の変化の記録になる）。スモーク実行後、直前コミットとのスクリーンショット差分に意図しない変化がないか目視確認する

スモークスクリプト自体が壊れた場合は、その修復を最優先タスクとして扱う（スモーク無しのままサイクルを重ねない）。

### 5. 記録
- tasks.md: タスクを DONE に移動し、完了メモを添える。**完了メモは2〜3文・200字程度まで**（何をどう解決したかの要点のみ。実装詳細・検証手順の記述はコミットメッセージと journal に委ね、DONE には書かない。長文メモは毎サイクルの読み込みコストになり、ループ全体を遅くする）
- docs/journal.md: エントリを1件追記する（書式は同ファイル冒頭を参照）
- 設計判断・ハマりどころ・次の自分への申し送りがあれば docs/insights.md を更新する
- プロジェクト固有の学び（失敗・効率化・設計パターン）は auto-memory にも保存する
- **軽微アーカイブはこのステップに畳み込む**: 記録追記の結果 journal.md・tasks.md が手順Mの閾値を超えたら、溢れた古い項目（journal は古いエントリ、tasks は番号の小さい DONE）を**今回の実装コミットに含めて** archive ファイルへ切り出す。この機械的な切り出しのために専用のメンテナンスサイクル・専用コミットを作らない。insights.md の統合だけは内容判断を伴うため、必要時に独立サイクルとして行ってよい

### 6. コミット
- 検証ゲートがすべて緑の状態でのみ `git commit` する（このループでは直接コミットが明示的に許可されている）
- コミットメッセージ: `feat({{COMMIT_SCOPE}}): T-xxx <内容>`（内容に応じて fix / refactor / chore / docs も使う）
- 検証が通らないままサイクルを終える場合はコミットせず、tasks.md の該当タスクを BLOCKED に移して原因を記録する

## タスクの受け入れ条件のルール

- **必須**: すべてのタスクの受け入れ条件は「動いている実物の上で確認できる状態」を含めて書く。「ユニットテストが通る」だけを受け入れ条件にしてはならない
- 純粋ロジック先行で作る場合は、**同一ゴール内に必ず配線タスクをペアで起票**し、配線タスクの受け入れ条件をスモークまたは実操作確認にする。配線されていないコードはマイルストーンの完了に数えない
- **UXの最低ライン**（プレースホルダー段階でも必須）: 画面に出る文言は利用者の言語（日本語）で読めること／その画面が何で・次に何をすればよいかが初見で分かること。これは「演出」ではなくコア要件であり、後回しにしない

## 手順G: ゴール設定サイクル（TODO が空のとき）

1. {{GOAL_DOC}}・docs/decisions.md・実装の現状・docs/journal.md の直近数件を確認する
2. 次のマイルストーン（ゴール）を1つ決める
   - **垂直スライスが未達成なら、垂直スライスの完成以外をゴールにしない**
   - {{GOAL_DOC}} に未実装の要素があれば、それを優先する
   - コア要素がすべて実装済みなら: リファクタリング／磨き込み／調整／コンセプトに合う新機能の発案 から選ぶ
3. ゴールを「1サイクルで終わる粒度 × 実物で確認できる受け入れ条件付き」のタスク群（目安3〜10個）に分解し、tasks.md の TODO に追加する
4. 新機能を発案した場合は、docs/decisions.md に「何を・なぜ・コンセプトとの整合」を追記する
5. ゴール設定の理由を journal.md に記録し、コミットして終了する（このサイクルでは実装しない）

### ゴール設定の制約
- 新機能は {{GOAL_DOC}} のコンセプトに合致するものだけ。方針を壊す要素（{{FORBIDDEN_FEATURES}}）は発案しない
- BLOCKED のタスクが残っている場合は、新ゴールより先にその解消・再定義を検討する
- 同系統のゴールが3回連続したら、次は別の系統を選ぶ

## 手順P: 初見レビューサイクル（{{REVIEW_INTERVAL}}サイクルごと・このサイクルでは実装しない）

無人ループは「タスクが完了し続けているのに、実物は壊れている」状態に陥りやすい。それを検出する唯一の手段が、初見の利用者になりきって実物を通しで触るこのサイクルである。

1. サーバー等をバックグラウンドで起動し、{{TARGET_ENV}} で成果物を**初見の利用者として**最初から通しで操作する（開発者として「動くはずの経路」をなぞるのではなく、画面に見えている情報だけを頼りに操作する）
2. 以下の観点で評価する:
   - 到達ラインに対して今どこまで来ているか
   - 初見で次に何をすればよいか分かるか（分からなければそれはバグと同格）
   - 表示崩れ・操作不能・違和感（放置すると次のNサイクルの土台が壊れたままになる）
   - {{GOAL_DOC}} の設計目標（{{DESIGN_PRIORITY}}）が実物に反映されているか
3. 気づきを重大度順に tasks.md へ起票する。**操作不能・表示崩れ級の問題は既存TODOより上（最優先）に置く**
4. スクリーンショット等の証跡を `docs/autopilot/shots/` に保存し、journal.md に手順Pのエントリを1件書いてコミットして終了する（起動したプロセスは必ず停止する）

## 手順PL: 計画ターン（ループ側が N 周ごとに差し込む・このターンでは実装しない）

ループの定期プランニング機能（dokodemo-claude のループ設定等）から `/dokodemo-claude-tools:autopilot-plan` が送られたときの手順。通常サイクルには数えず、強いモデルで進め方そのものを見直す。

1. tasks.md・journal.md の直近エントリ・insights.md・`git log --oneline -15` から、前回の計画以降の進捗と停滞を読み直す
2. 「到達ラインへの距離（垂直スライス未達のまま横展開していないか）」「BLOCKED の滞留・同種の失敗の繰り返し」「TODO の粒度・受け入れ条件・並び順」を評価する
3. 問題があれば tasks.md の TODO を組み替える（並び替え・分割・再定義・陳腐化タスクの削除・不足タスクの追加）。方針レベルの転換は docs/decisions.md に記録する
4. journal.md に計画エントリを1件追記し、記録ファイルのみの変更であることを確認してコミットして終了する（「変更なし」の判断も有効な結論として理由付きで記録する）

このターンではソースコードに触れない・サーバー等を起動しない。実物の確認は手順Pに委ねる。

## 手順M: メンテナンス（記録ファイルの肥大化防止）

記録ファイル（tasks.md・journal.md・insights.md）は毎サイクル読み返すため、無制限に増え続けるとコンテキストを圧迫する。ただし **journal・tasks の機械的なアーカイブは専用サイクルを設けず、手順5（記録）の中で実装コミットに畳み込む**（機械作業に1サイクル払うと、実装を1歩も進めないサイクルが積み重なりループ全体のペースを落とす。過去の実績で非実装サイクルが全体の約1/4を占めた）。insights の統合だけは内容判断を伴うため、閾値超過時に独立サイクルとして行ってよい。

### 肥大化の基準

アーカイブは実装コミットに畳み込むためコストがほぼゼロ。閾値を上げて溜め込むのではなく、**低い閾値でこまめに切り出し、毎サイクルの読み込み量を小さく保つ**。

- docs/journal.md が **150行を超える** → 手順5で切り出し（専用サイクル不要）
- docs/tasks.md の DONE 項目が **12件を超える** → 手順5で切り出し（専用サイクル不要）
- docs/insights.md が **100行を超える** → 独立のメンテナンスサイクルで統合（このサイクルでは実装しない）

### 対応
- **journal.md**: 冒頭の書式説明は残し、古いエントリから直近12件程度のみを残して溢れた分を docs/journal-archive.md の末尾に追記する（無ければ見出し付きで新規作成）。この切り出しは実装サイクルの手順5内で行い、実装と同じコミットに含める
- **tasks.md**: DONE セクションの古い項目から直近10件程度のみを残し、溢れた分を docs/tasks-archive.md の末尾に追記する。これも手順5内で実装コミットに含める
- **insights.md**: アーカイブせず、重複・陳腐化した項目を削除し、似た知見は1項目に統合して短くする（複数タスクにまたがる同一設計判断は経緯を捨てて最終結論のみに統合する）。これは内容判断を伴うため独立サイクルで行い、`chore({{COMMIT_SCOPE}}): insights整理` としてコミットしてよい
- アーカイブファイルは通常サイクルの「状態確認」では読まない（過去の経緯の調査時のみ参照）

## その他のルール

- 同じタスクが3サイクル連続で BLOCKED のままなら、正面突破をやめて代替アプローチを insights.md に記し、タスクを再定義する
- リポジトリを赤い状態（検証ゲートが失敗する状態）でコミットしない。やむを得ず中断する場合は未コミットのまま終え、次サイクルの復旧に委ねる
- エージェントによる実操作の検証は、数秒〜数十秒の判定窓に間に合わないことがある。リアルタイム性が要る検証で2回失敗したら実プレイで粘らず、テスタビリティフック（後述）や状態注入等の代替手段に切り替える
- 迷ったら「{{GOAL_DOC}} のコンセプトに近づくか」「実物を触る人の体験が良くなるか」「後から人間が読んで追跡できるか」を判断基準にする

## 実装フェーズの参考順序

**0. 検証基盤（最優先・省略禁止）** → 1. 垂直スライス → 2. 幅の拡張 → 3. 磨き込み・調整 → 4. QA

- **0. 検証基盤**: スモークスクリプトの整備と、テスタビリティフックの実装。テスタビリティフックとは、無人検証を可能にするための仕込み（例: {{TESTABILITY_HOOKS}}）。機能実装より先にこれを作る。検証手段のないまま積んだ機能は、壊れていても誰も気づけない
- **1. 垂直スライス**: {{VERTICAL_SLICE}} を最短で成立させる。見た目はプレースホルダーでよいが、UXの最低ライン（前述）は満たすこと
- **2. 幅の拡張**: コンテンツ・画面・機能の追加。垂直スライスが手順Pに合格してから
- **3. 磨き込み**: 見た目・演出・バランス・エラーハンドリング

この順序はあくまで参考であり、ゴール設定サイクルでの判断を優先してよい。ただし「0より先に1」「1より先に2」への着手は禁止。
